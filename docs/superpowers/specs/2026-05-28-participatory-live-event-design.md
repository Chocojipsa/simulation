# Participatory Live Event Design

## Goal

Change the portfolio demo from an isolated AI-only simulation into a shared live ticketing event where real browser users and AI participants compete in the same reservation flow.

The main story should be easy to explain:

```text
Many participants enter one ticketing event.
Some are real users in browsers.
Some are AI participants generated by the system.
All of them share the same Redis queue, PostgreSQL seat inventory, Kafka payment flow, and multi-server API layer.
```

This keeps the existing distributed simulation work useful, but moves the product surface closer to a realistic ticketing service.

## Scope

In scope:

- One active live event for the MVP.
- Human participants that join from the React frontend.
- AI participants that use the same public APIs as humans.
- Redis-backed waiting queue and live snapshot.
- PostgreSQL-backed seat hold and reservation state.
- Kafka-backed payment confirmation and result flow.
- A Korean frontend focused on the event room, seat map, queue, and participant progress.
- Local Docker deployment with nginx, two API servers, Redis, PostgreSQL, Kafka, worker, and AI traffic generator.
- Future production shape that can map to Lightsail or EC2 instances without changing application behavior.

Out of scope for this milestone:

- Real payment integration.
- Login, OAuth, or account management.
- Multiple concurrent public events.
- Admin-grade event management screens.
- Complex anti-bot or fraud prevention.
- Fully managed AWS infrastructure automation.

## Recommended Approach

Use a Live Event facade over the current simulation engine first, then rename internals gradually after behavior is stable.

The existing code already has useful boundaries:

- `simulation` for snapshots and participant state.
- `queue` for Redis admission.
- `seat` for PostgreSQL seat concurrency.
- `payment` for Kafka worker flow.
- `generator` for AI traffic.

Replacing all of that at once would slow the project down and increase regression risk. Instead, the next version should introduce event-facing APIs and UI while reusing current services internally where possible.

## Domain Model

### LiveEvent

`LiveEvent` is the public product concept. It represents one ticketing event.

Fields:

- `id`
- `title`
- `status`: `READY`, `COUNTDOWN`, `OPEN`, `ENDED`
- `opensAt`
- `endsAt`
- `seatCount`
- `aiParticipantCount`
- `createdAt`

MVP constraint:

- Only one active event exists at a time.
- The active event may be auto-created for local/demo profiles.

### Participant

Humans and AI should share one participant model.

Fields:

- `id`
- `eventId`
- `type`: `HUMAN`, `AI`
- `displayName`
- `status`
- `queueRank`
- `selectedSeatId`
- `seatSelectionAttempts`
- `paymentAttempts`
- `lastMessage`

Statuses:

- `WAITING_ROOM`: participant is present before entering the queue.
- `QUEUED`: participant clicked reserve and entered the Redis queue.
- `ADMITTED`: participant can attempt seat selection.
- `SELECTING_SEAT`: participant is actively trying to hold a seat.
- `PAYMENT_PENDING`: participant holds a seat and must confirm payment.
- `RESERVED`: reservation is complete.
- `PAYMENT_FAILED`: payment failed and the participant may retry if the event is still open.
- `FAILED`: participant cannot continue.

### Seat

PostgreSQL remains the authority for seat concurrency.

Fields:

- `eventId`
- `seatId`
- `label`
- `section`
- `row`
- `number`
- `status`
- `heldByParticipantId`
- `reservationId`

Statuses:

- `AVAILABLE`
- `HELD`
- `PAYMENT_IN_PROGRESS`
- `RESERVED`

## Architecture

```text
Browser users
  -> Vercel React frontend
  -> nginx or production load balancer
  -> api-a / api-b
  -> Redis queue and live snapshot
  -> PostgreSQL seat inventory
  -> Kafka payment topics
  -> worker

AI participants
  -> traffic-generator
  -> nginx or production load balancer
  -> api-a / api-b
  -> same Redis, PostgreSQL, Kafka, worker path
```

The important portfolio point is that AI participants do not bypass the system. They call the same reservation APIs that real users call, so the multi-server and concurrency behavior is visible.

## Event Flow

1. A live event exists in `READY` or `COUNTDOWN`.
2. A human opens the frontend and joins the active event.
3. The frontend stores the participant id in `localStorage`.
4. The event changes to `OPEN` after the countdown.
5. The user clicks `예약하기`.
6. The API places the participant into the Redis waiting queue.
7. Admission capacity allows a limited number of participants to leave the queue.
8. An admitted participant selects a seat.
9. The API attempts a PostgreSQL seat hold.
10. If the hold fails because another participant won the seat, the participant sees a conflict message and can select again.
11. If the hold succeeds, the participant moves to `PAYMENT_PENDING`.
12. Human participants click `결제 확인`.
13. AI participants automatically confirm payment after a small delay.
14. The API publishes a Kafka payment request.
15. The worker consumes the request and publishes a payment result.
16. The result listener updates PostgreSQL and Redis.
17. Successful payments become `RESERVED`.
18. Failed payments release the seat back to `AVAILABLE` and allow another participant to take it.

Payment should not wait for the entire queue to drain. Queue admission, seat selection, and payment processing should run concurrently.

## API Design

Event-facing endpoints:

```text
GET  /api/events/active
GET  /api/events/{eventId}/snapshot
POST /api/events/{eventId}/participants
POST /api/events/{eventId}/participants/{participantId}/queue
POST /api/events/{eventId}/participants/{participantId}/seats/{seatId}/hold
POST /api/events/{eventId}/participants/{participantId}/payment-confirm
POST /api/events/{eventId}/ai/start
```

Possible internal compatibility:

```text
eventId may map to the current simulationId while the migration is in progress.
```

This keeps the frontend and portfolio story event-oriented without forcing an immediate rename of every backend class.

## Redis Responsibilities

Redis should own fast-changing live state:

- Waiting queue.
- Queue rank.
- Participant live status.
- Seat map read model.
- Timeline/activity feed.
- API server request counters.
- Worker counters.
- Kafka pending/result counters.

Redis state must be event-scoped so local tests and future event expansion remain possible.

## PostgreSQL Responsibilities

PostgreSQL should own durable and concurrency-critical state:

- Event metadata.
- Seat inventory.
- Seat hold attempts.
- Reservations.
- Payment result application.

Seat hold should still be implemented with an atomic update or equivalent database-level guard. Redis may say a seat looks available, but PostgreSQL decides who actually wins.

## Kafka Responsibilities

Kafka should model the asynchronous payment boundary:

```text
payment.requested -> worker -> payment.result
```

The payment result should include:

- `eventId`
- `participantId`
- `seatId`
- `reservationId`
- `success`
- `failureReason`
- `handledBy`

The current planned payment failure rate should stay configurable. For portfolio readability, the default failure rate should be low, around 5%, so failed seats occasionally reopen without making the system look broken.

## Frontend Design

The first screen should be the event room, not a landing page.

Main areas:

- Header: event title, event status, countdown, participant count, reserved seat count.
- Seat map: visible seat inventory with status colors.
- My status panel: queue rank, selected seat, next action, payment confirm button.
- Activity feed: human and AI events in Korean.
- Infra panel: api-a/api-b counts, Redis queue size, Kafka pending/result counts, worker status.

Seat colors:

- Green: available.
- Yellow: held or payment pending.
- Gray: reserved.
- Blue: my selected seat.
- Red outline or message: conflict on last attempt.

The UI should make the concurrency visible without requiring the user to inspect logs.

## AI Participant Behavior

AI participants should act like fast but realistic users:

- Join the event through the public event API.
- Click reserve by calling the queue endpoint.
- Wait for admission.
- Pick a random currently available seat.
- Retry on seat conflict.
- Auto-confirm payment after a short delay.
- Stop after reservation success or after the event ends.

AI delay should remain configurable. A small delay is necessary so the frontend can show the event unfolding instead of jumping straight to the final state.

## Local Infrastructure

Local Docker should continue to run:

- `nginx`
- `api-a`
- `api-b`
- `traffic-generator`
- `worker`
- `redis`
- `kafka`
- `postgres`

The frontend should stay separate as a Vite app:

```text
frontend dev server -> /api proxy -> nginx:8080 -> api-a/api-b
```

This local shape matches the production concept while staying cheap and easy to run.

## Production Direction

The application should keep local and production profiles separate.

Likely low-cost first production shape:

- Frontend: Vercel.
- Backend: Lightsail instances or small EC2 instances.
- Database: AWS RDS PostgreSQL if budget allows, otherwise a self-hosted PostgreSQL instance for early demo only.
- Redis/Kafka: initially self-hosted on the backend or infra instance for cost control, with clear documentation that managed services are the scalable option.
- Load balancing: production LB can be added when budget allows. Local nginx still demonstrates the routing model.

The code should not assume that all services run in one container network. Service URLs and credentials must remain environment-driven.

## Error Handling

Expected domain failures should be visible in the event snapshot:

- Already queued.
- Still waiting in queue.
- Seat already selected by another participant.
- Seat no longer available.
- Payment failed.
- Event is not open.
- Event has ended.

Unexpected infrastructure failures can return 5xx in local development, but the UI should not treat normal race conditions as server errors.

## Testing Strategy

Backend unit tests:

- Human participant creation.
- Participant queue entry.
- Queue rank updates.
- Seat hold success.
- Seat conflict handling.
- Payment confirmation event publication.
- Payment success applies to PostgreSQL and Redis.
- Payment failure releases the seat.
- AI participant client uses public endpoints.

Frontend tests:

- Active event loading.
- Participant persistence through `localStorage`.
- Queue status rendering.
- Seat map color mapping.
- Payment confirm button state.
- Snapshot selector calculations.

Runtime verification:

- `docker compose up -d --build`
- `docker compose restart nginx`
- Start frontend dev server.
- Open multiple browser tabs.
- Start AI participants.
- Confirm that humans and AI share one queue and one seat map.
- Confirm that api-a and api-b both receive traffic.
- Confirm Kafka payment counters move.
- Confirm failed payment seats reopen and can be reserved by another participant.

## Migration Plan

1. Preserve current uncommitted payment/reopen/Kafka partition changes and finish integration verification.
2. Add event-facing API DTOs and controllers while reusing current simulation services internally.
3. Add human participant creation and browser participant persistence.
4. Update frontend from simulation dashboard to live event room.
5. Rework traffic generator wording and API use so AI participants join the active event.
6. Add event snapshot fields for queue, infra metrics, Kafka status, and participant activity.
7. Rename internal `Simulation` concepts only after the event-facing behavior is stable.

## Acceptance Criteria

- A user can open the frontend and join the active event.
- A user can click `예약하기` and see queue progress.
- Multiple browser tabs participate in the same event.
- AI participants participate in the same event as humans.
- Seat selection conflicts are visible.
- PostgreSQL prevents double reservation of one seat.
- Redis shows queue and live state.
- Kafka processes payment confirmation asynchronously.
- A payment failure releases the seat and another participant can reserve it.
- The frontend clearly shows multi-server, Redis, Kafka, and worker activity in Korean.
